## 第二部分 · 通讯协议规范

### 2.1 协议栈

```
Layer 7  业务层        Envelope JSON (自定义)
Layer 6  表示层        application/json; charset=utf-8
Layer 5  会话层        JWT Bearer + 卡密 + 硬件指纹
Layer 4  传输层        HTTP/1.1 or HTTP/2
Layer 3  TLS           TLS 1.2+ (强制 HTTPS, 自用期本地 HTTP 允许)
Layer 2  TCP           ...
```

**强制要求 (Phase 2+ 对外)**:
- 所有请求必须 HTTPS (Phase 1 自用期可 HTTP localhost)
- TLS 1.2 以下拒绝
- 证书: Let's Encrypt / DigiCert EV (Phase 4 购买)
- HSTS: `Strict-Transport-Security: max-age=31536000`

### 2.2 Envelope 响应格式 (强制)

**所有** `/api/client/*` 接口必须返回:

```json
// 成功
{
  "ok": true,
  "data": <业务载荷, 可以是 object / array / null>,
  "meta": {
    "trace_id": "a8f2c7d4",              // uuid4[:8]
    "ts": "2026-04-25T14:30:00+08:00",
    "server_version": "1.0.0",
    "request_id": "req_...",             // 客户端可选提供, 原样返回
    "cache": "miss" | "hit"              // 可选
  }
}

// 失败
{
  "ok": false,
  "error": {
    "code": "AUTH_401",                  // 必填, 见 2.3
    "message": "登录已过期, 请重新登录",   // 必填, 面向用户中文
    "hint": "点击右上角头像退出重登",      // 可选建议操作
    "details": {                         // 可选, 仅 DEBUG 模式返回
      "internal_error": "jwt.ExpiredSignatureError",
      "trace": "..."
    }
  },
  "meta": {
    "trace_id": "a8f2c7d4",
    "ts": "2026-04-25T14:30:00+08:00"
  }
}
```

**HTTP 状态码 vs Envelope 的 `ok`**:
- 业务错误: HTTP 200 + `ok: false` + `error.code=BUSINESS_*`
- 鉴权错误: HTTP 401/402/403 + `ok: false`
- 校验错误: HTTP 422 + `ok: false` + `error.code=VALIDATION_*`
- 服务器错误: HTTP 500 + `ok: false` + `error.code=INTERNAL_*`

### 2.3 错误码体系 (完整表)

| 错误码 | HTTP | 用户可读 message | 后端用途 | 客户端应对 |
|---|---|---|---|---|
| `AUTH_401` | 401 | 登录已过期, 请重新登录 | JWT 过期或无效 | 清 token, 跳登录页 |
| `AUTH_402` | 402 | 您的套餐已到期, 请续费 | 卡密过期 | 跳订阅中心 |
| `AUTH_403` | 403 | 此功能需升级到高级版 | FEATURE 不匹配 | 显锁定卡 + 升级按钮 |
| `AUTH_423` | 423 | 账号已被锁定, 请联系客服 | 风控触发 | 锁 UI + 客服入口 |
| `AUTH_498` | 498 | 登录设备已变更, 请重新登录 | 硬件指纹不匹配 | 重新激活卡密 |
| `VALIDATION_422` | 422 | 请求参数不正确, 请检查输入 | Pydantic 校验 | 提示用户检查表单 |
| `RESOURCE_404` | 404 | 请求的资源不存在 | 资源缺失 | 显空状态 |
| `CONFLICT_409` | 409 | 操作冲突, 请刷新后重试 | 乐观锁 / 重复创建 | 自动重新拉取 + 重试 |
| `RATE_LIMIT_429` | 429 | 请求过于频繁, 请稍后再试 | 限流 | 显 cooldown 倒计时 |
| `INTERNAL_500` | 500 | 服务器内部错误, 请稍后重试 | 未捕获异常 | 上报 + 重试 1 次 |
| `UPSTREAM_502` | 502 | 上游服务暂时不可用 | MCN/快手故障 | 显"系统维护, 稍后重试" |
| `MAINTENANCE_503` | 503 | 系统维护中 | 计划维护 | 显维护公告 |
| `GATEWAY_TIMEOUT_504` | 504 | 请求超时, 请重试 | 上游超时 | 自动重试 1 次 |
| `BUSINESS_NO_URL` | 200 | 该剧暂无可用素材, 请稍后 | 无 URL | 提示换剧 |
| `BUSINESS_BLACKLIST` | 200 | 该剧已被屏蔽, 无法发布 | 黑名单命中 | 剧列表置灰 |
| `BUSINESS_MCN_80004` | 200 | 账号暂无法发布该剧, 系统自动切换 | MCN 80004 | 静默切换 |
| `BUSINESS_QUOTA_EXCEEDED` | 200 | 今日发布额度已用完 | 配额 | 显剩余额度 + 续费 |
| `BUSINESS_ACCOUNT_FROZEN` | 200 | 该账号已暂停, 请先解除 | 账号 frozen | 跳账号详情 |
| `BUSINESS_INVALID_COOKIE` | 200 | 账号登录状态已失效, 请重新登录 | cookie 无效 | 触发重登流程 |

**错误 message 硬性规则** (v3 §18.3):
- ❌ **禁止** 出现 `sqlite3.OperationalError` / `cookie invalid` / `breaker triggered` / `sig3 reject` 等技术原文
- ❌ **禁止** 堆栈信息 / 数据库字段名 / 文件路径
- ✅ **必须** 用户中文 + 动作建议

### 2.4 Rate Limiting (限流)

按用户 + 按 IP 双维度:

```
用户维度 (JWT sub):
  /api/client/auth/*              5 req/min    (防爆破)
  /api/client/publish/batch       10 req/min   (防刷)
  /api/client/*                   120 req/min  (通用)
  心跳 /api/client/auth/heartbeat 免限流

IP 维度 (防未登录攻击):
  /api/client/*                   60 req/min per IP

超限响应:
  HTTP 429 + error.code=RATE_LIMIT_429
  Header: Retry-After: <秒>
  Header: X-RateLimit-Limit: 120
  Header: X-RateLimit-Remaining: 0
  Header: X-RateLimit-Reset: 1720000000
```

实现: `slowapi` (Python) or `nginx limit_req_zone` (生产).

### 2.5 Trace ID 链路追踪

```
每请求生成: trace_id = uuid4().hex[:8]

Middleware 设置:
  request.state.trace_id = trace_id
  logger.contextvars.bind(trace_id=trace_id)
  response.headers["X-Trace-ID"] = trace_id

所有 envelope.meta.trace_id 自动带上
所有 error.meta.trace_id 必带
日志每行含 [trace_id=xxxxxxxx]

客户端反馈时带 trace_id, 后端可 10 秒内定位问题
```

### 2.6 幂等性 (Idempotency)

写操作 (POST/PUT/DELETE) 建议客户端带 `Idempotency-Key` header:

```
POST /api/client/publish/batch
Idempotency-Key: <uuid4>

第一次: 200 + 业务结果
第二次 (相同 key 10 分钟内): 200 + 缓存的业务结果 (不重复执行)
```

服务端实现: Redis `SETNX key:{idempotency_key} EX 600`.

**强制幂等**: `/publish/batch`, `/publish/manual`, `/accounts/add`, `/accounts/{id}/*`, `/autopilot/start`.

### 2.7 时间格式

- 所有时间: **ISO 8601 with timezone** (`2026-04-25T14:30:00+08:00`)
- 禁止: Unix timestamp (客户端易混乱) / `YYYY-MM-DD HH:MM:SS` 无时区
- 服务器时区: **Asia/Shanghai (UTC+8)**
- Request header 建议: `X-Client-Timezone: Asia/Shanghai`

### 2.8 分页约定

```
GET /api/client/accounts?page=1&size=50

响应 data:
{
  "items": [...],
  "pagination": {
    "page": 1,
    "size": 50,
    "total": 342,
    "total_pages": 7,
    "has_next": true
  }
}

限制:
  size 最大 200 (超出 422)
  page 从 1 开始
  默认 size=20
```

### 2.9 排序与筛选约定

```
?sort=created_at.desc,score.asc         多字段排序, . 分隔方向
?filter=status:active,tier:pro          简单筛选, 逗号分隔 key:value

```

### 2.10 编码与字符集

- 请求/响应: UTF-8
- URL query 参数: URL encoded
- Body: `application/json; charset=utf-8`
- 文件上传: `multipart/form-data` (仅 cookie 导入等少数场景)

## 第三部分 · 认证与授权

### 3.1 激活 + 登录完整流程

```
[首次激活 — 客户端 exe 首启]
  │
  ├─ 用户输入卡密 + 手机号
  │
  ├─ 客户端计算硬件指纹:
  │    fingerprint = sha256(cpu_id + motherboard_sn + disk_serial)[:32]
  │
  ├─> POST /api/client/auth/activate
  │   body: { license_key, phone, fingerprint, client_version, os_info }
  │
  └─ 服务器:
     1. 校验 license_key 存在 + 未过期 + 未绑定其他指纹
     2. 绑定: license.device_fingerprint = fingerprint
     3. 创建 user (如不存在)
     4. 签发 JWT + refresh_token
     5. 返: { token, refresh_token, user, plan, expires_at }
  │
[后续登录 — 同设备]
  │
  ├─> POST /api/client/auth/login
  │   body: { phone, password, fingerprint }
  │
  └─ 服务器:
     1. 校验密码
     2. 校验 fingerprint == license.device_fingerprint
     3. 签 JWT + refresh_token
  │
[心跳 — 每 5 分钟]
  │
  ├─> POST /api/client/auth/heartbeat
  │   Authorization: Bearer <JWT>
  │   body: { fingerprint }
  │
  └─ 服务器:
     1. verify JWT (若过期 → 401)
     2. 校验 license.expires_at > now
     3. 校验 fingerprint 仍匹配
     4. 返 { ok, server_time, plan, expires_at }
  │
[Token 过期 — 客户端收到 401]
  │
  ├─> POST /api/client/auth/refresh
  │   body: { refresh_token, fingerprint }
  │
  └─ 服务器:
     1. verify refresh_token
     2. blacklist 老 refresh_token (Redis)
     3. 签新 JWT + 新 refresh_token
```

### 3.2 JWT 结构

```json
{
  "sub": "user_12345",              // user id
  "license_id": "lic_abc",
  "fingerprint_hash": "xxx",         // 硬件指纹 hash (防盗用)
  "plan": "pro",                     // manual | pro | team
  "role": "owner",                   // owner | admin | operator | viewer
  "features": [                      // FEATURE_* 列表
    "FEATURE_MANUAL_BATCH", "FEATURE_AI_PICK", ...
  ],
  "iat": 1720000000,
  "exp": 1720001800,                 // 30 分钟有效 (短 token)
  "iss": "ks-matrix",
  "aud": "ks-client"
}
```

**密钥管理**:
- HS256 + 服务端密钥 (`.env` 环境变量)
- 密钥每 3 个月轮换 (Phase 2+)
- 轮换期间: 同时支持新旧密钥 verify

### 3.3 Refresh Token

```json
{
  "sub": "user_12345",
  "license_id": "lic_abc",
  "token_type": "refresh",
  "iat": 1720000000,
  "exp": 1722592000,                 // 30 天有效
  "jti": "uuid"                      // 用于 blacklist
}
```

- **一次性**: 用过即 blacklist (Redis `SETEX blacklist:{jti} 2592000 1`)
- 短 JWT + 长 refresh 策略减少窃取风险

### 3.4 权限校验流程

```python
# dependencies.py

async def jwt_license_auth(request: Request) -> UserContext:
    """客户端 JWT + 卡密双验"""
    # 1. 提取 JWT
    token = extract_bearer(request)
    if not token:
        raise HTTPException(401, detail={"code": "AUTH_401"})
    
    # 2. verify JWT
    try:
        payload = jwt.decode(token, SECRET, algorithms=["HS256"])
    except jwt.ExpiredSignatureError:
        raise HTTPException(401, detail={"code": "AUTH_401"})
    except jwt.InvalidTokenError:
        raise HTTPException(401, detail={"code": "AUTH_401"})
    
    # 3. 卡密校验 (Redis 缓存 60s)
    license = await get_license_cached(payload["license_id"])
    if not license or license.expires_at < now():
        raise HTTPException(402, detail={"code": "AUTH_402"})
    
    # 4. 硬件指纹校验
    fp_hash = hashlib.sha256(
        request.headers.get("X-Device-Fingerprint", "").encode()
    ).hexdigest()
    if fp_hash != payload["fingerprint_hash"]:
        raise HTTPException(498, detail={"code": "AUTH_498"})
    
    # 5. blacklist 检查
    if await redis.exists(f"blacklist:{payload.get('jti')}"):
        raise HTTPException(401, detail={"code": "AUTH_401"})
    
    return UserContext(
        user_id=payload["sub"],
        plan=payload["plan"],
        role=payload["role"],
        features=set(payload["features"]),
        license=license,
    )


def require_feature(feature: str):
    """装饰器: 要求用户拥有特定 FEATURE"""
    async def dep(user: UserContext = Depends(jwt_license_auth)):
        if feature not in user.features:
            raise HTTPException(403, detail={
                "code": "AUTH_403",
                "message": "此功能需升级到高级版",
                "hint": "前往订阅中心升级"
            })
        return user
    return dep


def require_role(role: str):
    """装饰器: 要求特定角色 (owner/admin/...)"""
    async def dep(user: UserContext = Depends(jwt_license_auth)):
        if ROLE_HIERARCHY[user.role] < ROLE_HIERARCHY[role]:
            raise HTTPException(403, detail={"code": "AUTH_403"})
        return user
    return dep
```

## 第五部分 · 数据脱敏规则 (关键)

### 5.1 字段黑名单 (客户端 API 绝不返回)

任何 `/api/client/*` 响应必须经过 `sanitize()` 函数过滤以下字段:

```python
# dashboard/api_app/services/sanitize.py

INTERNAL_FIELDS_BLACKLIST = {
    # 任务内部标识
    "task_id", "batch_id_internal", "worker_name", "thread_id",
    # 算法参数
    "recipe", "image_mode", "ffmpeg_params", "kirin_mode6",
    "zhizun_mode5", "dedup_strength", "process_mode",
    # 熔断器
    "breaker_name", "breaker_state", "consec_fails",
    "cooldown_sec", "threshold", "half_open_at",
    # 快手协议
    "sig3", "api_ph", "cookie", "token", "photo_id_int",
    # LLM
    "llm_provider", "llm_model", "prompt", "confidence_score",
    "llm_cost_cents", "strategy_code",
    # 数据库
    "numeric_uid", "device_serial", "db_id",
    # 错误原文
    "raw_error_stack", "db_error_code", "internal_exception",
    # 审计
    "api_key", "webhook_secret", "fingerprint",
}

def sanitize(data: Any, role: str = "client") -> Any:
    """递归过滤字段"""
    if isinstance(data, dict):
        return {k: sanitize(v) for k, v in data.items()
                if k not in INTERNAL_FIELDS_BLACKLIST
                and not k.startswith("_")}
    elif isinstance(data, list):
        return [sanitize(x) for x in data]
    else:
        return data
```

**强制应用**: 所有 `/api/client/*` response_model 中间件自动 sanitize.

### 5.2 字段翻译规则

| 内部字段 | 客户端视图 | 映射 |
|---|---|---|
| `tier: "testing"` | `status: "观察期"` | TIER_TO_STATUS_MAP |
| `tier: "viral"` | `status: "爆款期"` | |
| `tier: "frozen"` | `status: "已冻结"` | |
| `recipe: "kirin_mode6"` | `dedup_preset: "标准"` | RECIPE_TO_PRESET_MAP |
| `recipe: "zhizun_overlay"` | `dedup_preset: "强力"` | |
| `confidence_score: 0.87` | `confidence: "高"` | >0.8 = 高, >0.5 = 中, else = 低 |
| `breaker_state: "open"` | (不返) 仅在 exceptions 里翻译为 "系统自动保护中" | |
| `error: "sig3 reject"` | `failure_reason: "发布协议异常, 系统将自动修复"` | |

### 5.3 特殊敏感数据处理

```
手机号:     13337289759  →  133****9759
身份证:     xxx          →  (禁止返回)
Cookie:     xxxxxxx      →  (禁止返回, 仅后端用)
IP 地址:    192.168.1.100 → 192.168.1.*** (审计日志脱敏)

## 第七部分 · 安全要求

### 7.1 身份认证安全

- JWT 密钥: 随机 64 字节, 环境变量, 每 3 月轮换
- Refresh Token: 一次性, 用完 blacklist
- 密码: bcrypt (cost=12), 永不明文存储
- 卡密: 硬件指纹绑定 + 后端白名单验证

### 7.2 防重放攻击

```
Phase 2+: 客户端请求带 nonce + timestamp
  X-Nonce: <random 16 bytes>
  X-Timestamp: <unix seconds>
  X-Signature: HMAC-SHA256(secret, method + path + body + nonce + ts)

服务端:
  ts 偏差 > 5 分钟 → 拒 (429)
  nonce 已用过 (Redis SETNX 10min) → 拒
  签名不匹配 → 拒 (401)
```

### 7.3 SQL 注入防护

```
✅ 必须参数化:
  cursor.execute("SELECT * FROM accounts WHERE id = ?", (account_id,))

❌ 禁止字符串拼接:
  cursor.execute(f"SELECT * FROM accounts WHERE id = {account_id}")

工具:
  pre-commit: detect-sql-injection (自定义 hook)
  ruff: 'S608' 规则
  code review: PR 必查
```

### 7.4 XSS / CSRF

- 本项目前端是 PyQt6 桌面端, **无 XSS 风险**
- **CSRF**: 所有写操作要 JWT (即 Bearer token), 不依赖 Cookie → 无 CSRF 风险
- 若开发辅助 Web 页 (订阅页等): 启用 `httponly` cookie + `SameSite=Strict`

### 7.5 DDoS 防护

```
Phase 2+:
  CloudFlare 前置 (免费 plan 够用)
  开启 "I'm Under Attack" 模式开关
  API 速率限制 (见 2.4)
  IP 黑名单: fail2ban 监控 /api/client/auth/*, 5 次失败 ban 1 小时
```

### 7.6 审计日志完整性

```
所有以下事件必须写 audit_logs 表:
  - 登录成功/失败

## 第九部分 · 客户端-服务器交互时序

> 本节**只描述协议时序**, 不描述 PyQt6 如何实现.

### 9.1 启动期 (冷启动 → 进入首页)

```
[客户端]                            [服务器]
  │
  ├─ 1. 本地读 config + token cache
  │
  ├─ 2. 若 token 有效:
  │    GET /api/client/license/status
  │    → 200 {plan, features, expires}    [2s 超时]
  │    若 401: 走 refresh or 重登
  │
  ├─ 3. GET /api/client/overview
  │    → 200 {today, trend, ...}          [2s 超时]
  │
  ├─ 4. 并行: 预取常用数据
  │    GET /api/client/accounts (size=20)
  │    GET /api/client/exceptions (limit=10)
  │
  └─ 5. 启动心跳定时器 (5 min)
```

### 9.2 运行期 (用户浏览 + 操作)

```
[用户点"账号"菜单]
  │
  ├─ 1. GET /api/client/accounts?page=1&size=50
  │    → 200 (30s cache)
  │
  ├─ 2. 并行: GET /api/client/accounts/groups
  │
[用户点某账号]
  │
  ├─ 3. GET /api/client/accounts/{id}
  │
[用户点"批量暂停"]
  │
  ├─ 4. 二次确认
  │
  ├─ 5. POST /api/client/accounts/batch
  │    Idempotency-Key: <uuid>
  │    body: { ids, action: "pause" }
  │    → 200 {affected: 5, failed: []}
  │
  ├─ 6. 失效 cache, 重新拉取列表
  │    GET /api/client/accounts?page=1 (cache bypass)

[后台心跳]
  │
  └─ 每 5 min:
     POST /api/client/auth/heartbeat
     → 200 {server_time, expires, notice?}
     若 401: 走 refresh
     若 402: 弹"套餐过期"对话框, 跳订阅
```

### 9.3 离线期 (网络断开)

```
策略:
  1. 所有 GET 请求先读本地 cache, 失败时显示 "离线中" 提示
  2. 所有 POST/PUT/DELETE 请求缓存到本地 outbox
  3. 网络恢复后, outbox 按序 replay (带 Idempotency-Key)
  4. 超过 30 分钟无心跳 → 锁定 UI, 要求重连

cache 策略:
  - license_status:      本地 encrypted JSON, 24h 有效
  - overview:            10s SWR (stale while revalidate)
  - accounts (列表):     30s SWR
  - dramas:              5min SWR

禁止离线 30 分钟以上使用客户端:
  ├─ 心跳失败 3 次连续 → 显示"网络问题, 正在重连"
  ├─ 心跳失败 5 分钟累计 → 禁用写操作, 仅可读 cache
  └─ 心跳失败 30 分钟累计 → 锁定界面, 必须重连
```

### 9.4 异常期 (错误重试策略)

```
网络错误 (connect timeout, DNS fail):
  ├─ 1 秒后重试, 最多 3 次
  ├─ 仍失败 → 显示 "网络超时", 提供 "重试" 按钮
  └─ 手动重试前不继续自动重试

5xx 错误:
  ├─ 指数退避重试 (1s, 2s, 4s), 最多 3 次
  └─ 仍失败 → 显示"服务暂不可用", 上报 trace_id

4xx 错误 (非 401):
  ├─ 不重试, 立即显示 error.message
  └─ 上报到客户端日志 (含 trace_id)

401 AUTH_401:
  ├─ 自动 POST /auth/refresh
  ├─ 成功: 重试原请求
  └─ 失败: 清 token, 跳登录页

402 AUTH_402:
  ├─ 弹对话框 "套餐已过期"
  └─ 跳订阅中心

403 AUTH_403:
  ├─ 显示 FeatureLockCard
  └─ "升级到 X 版本" 按钮跳订阅

429 RATE_LIMIT_429:
  ├─ 显示 "请稍后再试" + 倒计时
  ├─ 倒计时结束后自动允许再次点击
  └─ 使用 Retry-After header

超时 (gateway 504):
  └─ 同 5xx 重试策略
```

### 9.5 退出期

```
用户点"退出":
  ├─ 1. POST /api/client/auth/logout (best effort, 2s 超时)
  ├─ 2. 清 local token + cache
  └─ 3. 关闭客户端
```

### 10.3 错误处理模式

```python
# routers/client/publish.py

@router.post("/batch")
async def publish_batch(
    body: PublishBatchIn,
    user: UserContext = Depends(jwt_license_auth),
) -> Envelope[PublishBatchOut]:
    try:
        result = await publish_facade.create_batch(user, body)
        return Envelope.ok(result)
    
    except QuotaExceededError as e:
        return Envelope.error("BUSINESS_QUOTA_EXCEEDED",
                              "今日发布额度已用完",
                              hint="升级套餐可获更大额度")
    
    except AccountFrozenError as e:
        return Envelope.error("BUSINESS_ACCOUNT_FROZEN",
                              f"账号 {e.nickname} 已暂停, 请先解除",
                              hint="前往账号管理解除暂停")
    
    except Exception as e:
        # 未预期错误: 记日志 + 返 500
        logger.exception(f"publish_batch failed: {e}")
        return Envelope.error("INTERNAL_500",
                              "服务器内部错误, 请稍后重试")
```

## 第十一部分 · Phase 2+ 服务端演进

### 11.1 演进路线

```
Phase 1 (现在): 单机 FastAPI + SQLite
  - 100% 自用 / 内测 <10 人
  - 延迟 <200ms
  - 无 HTTPS (本地)

Phase 2: 云端 FastAPI + PG + Redis
  - 内测 50-100 人
  - CloudFlare + Let's Encrypt
  - 监控 + 日志聚合
  - 业务算法部分挪云

Phase 3: 多实例 + 水平扩展
  - 付费 200-500 人
  - Nginx 负载均衡
  - PG 主从 + 读写分离
  - 定时任务 Celery

Phase 4: 全套加固上线
  - 付费 500-2000 人
  - 客户端 Nuitka + VMProtect + Themida
  - DigiCert EV 证书
  - 多 region 部署

Phase 5: Go 重写 + 大规模
  - 付费 2000+
  - FastAPI → Go (Gin)
  - PG Cluster + Redis Cluster
  - K8s + GSLB
```

### 11.2 数据库迁移 (SQLite → PostgreSQL)

```
触发: 付费用户 > 100 or 每日写入 > 100万条

迁移步骤:
  1. 用 pgloader 导数据 (SQLite → PG)
  2. 改 SQLAlchemy URL (env)
  3. 测试: 全契约测试 + 压测
  4. 灰度: 先 10% 流量到 PG
  5. 全量: 100% 后关闭 SQLite

兼容性:
  DBManager 抽象, 切换只改连接串
  SQL 方言: SQLite 子集 (保证兼容), 必要时才用 PG 特性
```

### 11.3 服务端算法迁移 (§3.1 产品总体规划)

```
当前: core.* 算法在客户端本地 (PyQt6 exe 内)
Phase 2+: 关键算法挪到服务端

必挪:
  candidate_builder (选剧算法)      → /api/client/schedule/next
  match_scorer (账号×剧打分)        → 服务端内嵌
  burst detection (爆款检测)        → 云端数据聚合更好
  sig3 签名 (现有已在 MCN 代签)     → 服务端代签

可选挪:
  dedup (去重处理)                   → 仍客户端 (带宽成本)
  ffmpeg 视频处理                    → 仍客户端 (GPU/CPU)

客户端"瘦身":
  Phase 2-3: core/ 代码仍打包, 但关键函数走 API
  Phase 4+: core/ 中敏感算法用 "空壳" 替换, 实际走 API
```

### 11.4 代签层架构 (Phase 3+)

```
客户端:
  POST /api/client/publish/sign
  body: { body_to_sign: "..." }
  → { signed: "...", expires_at: "..." }

服务端:
  从 MCN :50002 获取 sig3
  缓存 + 返回
  客户端拿 sig3 自己发 cp.kuaishou.com

优势:
  - 签名算法不落地客户端 (反破解)
  - 可统一升级 (快手反爬更新 48h 内)
  - 限流 + 风控
```
