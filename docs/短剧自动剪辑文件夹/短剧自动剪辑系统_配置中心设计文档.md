# 短剧自动剪辑系统_配置中心设计文档

## 1. 文档目标

本文档用于定义“短剧自动剪辑系统”的配置中心设计方案，覆盖：

- 页面分区与交互结构
- AI 引擎角色拆分
- 解析、理解、评分、编排、导出等策略参数
- 默认配置建议
- 配置存储建议

配置中心不是单纯的“填 API Key 页面”，而是：

**AI 能力、规则策略、导出风格、人工兜底的统一控制台。**

---

## 2. 配置中心在系统中的作用

对这套短剧自动剪辑系统来说，配置中心承担四个核心职责：

1. 管理 AI Provider、模型与密钥
2. 管理剧情理解与自动编排策略
3. 管理导出样式、人工审核与失败兜底规则
4. 管理任务并发、超时与中间产物保留策略

如果没有配置中心，系统后续会出现以下问题：

- 不同步骤只能共用一个模型，成本难控
- Prompt 无法版本化，调优容易混乱
- 编排风格无法快速切换
- 人工兜底与自动化边界不清晰

---

## 3. 页面结构建议

建议配置中心采用：

- 左侧二级导航
- 右侧配置内容区
- 顶部环境状态条
- 底部“保存 / 测试 / 回滚默认值”操作区

推荐分为 8 个一级分区：

1. `AI 引擎配置`
2. `Prompt 模板配置`
3. `解析流程配置`
4. `剧情理解配置`
5. `评分与编排配置`
6. `导出与包装配置`
7. `人工审核配置`
8. `系统任务配置`

---

## 4. AI 引擎配置

### 4.1 设计原则

AI 不能只配置“一个默认模型”，必须按角色拆开。

原因：

- 不同步骤的成本敏感度不同
- 不同步骤对速度要求不同
- 不同步骤的稳定性和输出形式不同
- 某一个模型失效时，需要单独切换或回退

### 4.2 AI 角色拆分

建议拆成以下 6 个角色：

1. `转写模型`
2. `段摘要模型`
3. `剧情理解模型`
4. `主题线发现模型`
5. `编排决策模型`
6. `文案生成模型`

### 4.3 Provider 配置字段

每个 Provider 建议包含：

- `provider_name`
- `enabled`
- `endpoint`
- `api_key`
- `default_model`
- `timeout_sec`
- `retry_times`
- `rpm_limit`
- `health_status`
- `last_tested_at`

### 4.4 角色绑定字段

每个角色应支持单独配置：

- `provider`
- `model`
- `fallback_provider`
- `fallback_model`
- `temperature`
- `max_tokens`
- `timeout_sec`
- `retry_times`
- `json_mode`

## 5. AI 角色详细配置建议

### 5.1 转写模型

用途：

- 音频转写
- 句级时间戳
- 可选说话人标签

建议字段：

- `enabled`
- `provider`
- `model`
- `language`
- `with_timestamps`
- `with_speaker_labels`
- `max_audio_minutes_per_chunk`
- `fallback_to_subtitle_file`
- `timeout_sec`
- `retry_times`

推荐默认值：

```json
{
  "enabled": true,
  "provider": "openai",
  "model": "gpt-4o-mini-transcribe",
  "language": "zh",
  "with_timestamps": true,
  "with_speaker_labels": false,
  "max_audio_minutes_per_chunk": 20,
  "fallback_to_subtitle_file": true,
  "timeout_sec": 300,
  "retry_times": 2
}
```

### 5.2 段摘要模型

用途：

- 给剧情段生成一句简洁摘要

建议字段：

- `provider`
- `model`
- `temperature`
- `max_tokens`
- `summary_style`
- `output_json`
- `retry_times`

推荐默认值：

```json
{
  "provider": "openai",
  "model": "gpt-5.4-mini",
  "temperature": 0.2,
  "max_tokens": 300,
  "summary_style": "short_event_summary",
  "output_json": true,
  "retry_times": 2
}
```

### 5.3 剧情理解模型

用途：

- 判断剧情段属于背景、冲突、反转、高潮、悬念
- 识别情绪类型与角色关系变化

建议字段：

- `provider`
- `model`
- `temperature`
- `max_tokens`
- `label_schema_version`
- `use_segment_context_window`
- `context_window_size`
- `strict_json_mode`

推荐默认值：

```json
{
  "provider": "openai",
  "model": "gpt-5.4",
  "temperature": 0.1,
  "max_tokens": 800,
  "label_schema_version": "v1",
  "use_segment_context_window": true,
  "context_window_size": 3,
  "strict_json_mode": true
}
```

### 5.4 主题线发现模型

用途：

- 聚合主题线
- 形成“女主逆袭线”“婚姻冲突线”等运营视角

建议字段：

- `provider`
- `model`
- `temperature`
- `max_theme_count`
- `min_segments_per_theme`
- `allow_custom_theme_name`
- `merge_similar_theme_lines`

推荐默认值：

```json
{
  "provider": "openai",
  "model": "gpt-5.4",
  "temperature": 0.2,
  "max_theme_count": 5,
  "min_segments_per_theme": 3,
  "allow_custom_theme_name": true,
  "merge_similar_theme_lines": true
}
```

### 5.5 编排决策模型

用途：

- 主力 A / 主力 B 自动编排
- 辅助素材模板编排
- 桥接文案生成
- 断点选择

建议字段：

- `provider`
- `model`
- `temperature`
- `max_tokens`
- `template_family`
- `enforce_duration_limit`
- `allow_bridge_text`
- `allow_segment_compression`
- `main_a_strategy`
- `main_b_strategy`
- `aux_generation_count`

推荐默认值：

```json
{
  "provider": "openai",
  "model": "gpt-5.4",
  "temperature": 0.3,
  "max_tokens": 1500,
  "template_family": "mount_conversion_v1",
  "enforce_duration_limit": true,
  "allow_bridge_text": true,
  "allow_segment_compression": true,
  "main_a_strategy": "mainline_conversion",
  "main_b_strategy": "theme_focus",
  "aux_generation_count": 6
}
```

## 6. Prompt 模板配置

### 6.1 设计原则

Prompt 是高频调优对象，必须单独做版本化管理。

### 6.2 建议配置项

- `segment_summary_prompt_version`
- `story_understand_prompt_version`
- `theme_line_prompt_version`
- `main_a_prompt_version`
- `main_b_prompt_version`
- `aux_prompt_version`
- `copy_prompt_version`

### 6.3 页面能力建议

- 查看当前生效版本
- 一键切换版本
- 复制成新版本
- 查看最后修改时间
- 查看回滚历史

## 13. 页面布局建议

建议配置中心页面结构如下：

### 左侧导航

- AI 引擎
- Prompt 模板
- 解析策略
- 剧情理解
- 评分编排
- 导出包装
- 人工审核

#### 顶部：环境与状态条

- 当前环境
- 当前生效配置集
- Provider 总状态
- 最后保存时间

#### 中部：配置卡片区

按分区显示多个配置卡片，例如：

- Provider 列表
- 角色绑定
- Prompt 版本
- 权重参数
- 导出参数

#### 底部：操作区

- `保存配置`
- `测试当前 Provider`
- `恢复默认值`
- `导出配置`

## 14. 配置存储建议

### 14.1 V1 建议

V1 不强求单独建很多配置表，可以先用一个集中式配置表保存。

推荐表结构思路：

- `key`
- `config_type`
- `value_json`
- `updated_at`
- `updated_by`

例如：

- `highlight_ai_engines`
- `highlight_prompt_versions`
- `highlight_parse_policy`
- `highlight_story_policy`
- `highlight_score_policy`
- `highlight_export_policy`

### 14.2 后续演进

如果配置越来越复杂，再拆分成：

- `highlight_providers`
- `highlight_model_bindings`
- `highlight_prompt_versions`
- `highlight_runtime_policies`

## 15. V1 最小可用配置集

如果只做第一版，建议至少支持下面这些核心配置：

### 必做

- AI Provider 列表
- 6 个角色绑定
- Prompt 版本切换
- 主力 / 辅助时长
- 断点评分权重
- 重复惩罚阈值
- 导出比例与字幕开关
- 主力素材是否必须人工审核

### 可后置

- 多环境配置切换
- 配置历史回滚
- 导出配置包
- 团队协作与权限管理

---

## 16. 最终建议

配置中心设计的核心，不是让用户“填写参数”，而是让系统后续具备：

- 可切模型
- 可切 Prompt
- 可切编排风格
- 可控成本
- 可控人工兜底边界

一句话总结：

**配置中心不是技术设置页，而是短剧自动剪辑系统的内容生产策略中心。**
