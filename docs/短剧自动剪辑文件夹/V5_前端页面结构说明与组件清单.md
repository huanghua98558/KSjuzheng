## 3. 开发落地总原则

### 3.1 不推翻现有 web 工程

建议保留现有：

- `main.tsx`
- `App.tsx`
- `Layout.tsx`
- `lib/api.ts`
- `lib/auth.ts`
- 基础 `ui` 组件

原因：

- 当前登录、路由、自动刷新、事件流 toast 已经存在
- 这些是可直接复用的壳层能力
- 现在最值钱的是重建页面与业务组件，而不是重搭基础设施

### 3.2 先新增 V5 页面，不马上删旧页面

建议采用双轨策略：

第一阶段：

- 保留现有页面
- 新增 V5 页面与 V5 组件
- 路由先挂在新路径下

第二阶段：

- 验证 V5 页面可用
- 再把旧页面替换成 V5

这样风险最低。

### 3.3 先做页面壳层和数据骨架，再做细节交互

开发顺序建议：

1. 搭全局工业后台布局
2. 搭 10 个页面骨架
3. 接通列表和概览数据
4. 再补弹窗、批量操作、抽屉、快捷键
5. 最后做细节状态和视觉微调

## 6. 推荐目录结构

为了避免把所有代码继续堆在 `pages/`，建议从 V5 开始做轻量结构升级。

## 6.1 推荐结构

```text
web/src/
├─ app/
│  ├─ routes.tsx
│  └─ queryClient.ts
├─ components/
│  ├─ ui/                     继续保留现有 Button/Card/Tabs 等
│  ├─ layout/
│  ├─ production/
│  ├─ tables/
│  ├─ panels/
│  └─ feedback/
├─ features/
│  ├─ console/
│  ├─ intake/
│  ├─ pipeline/
│  ├─ assets/
│  ├─ project/
│  ├─ editing/
│  ├─ review/
│  ├─ delivery/
│  ├─ configs/
│  └─ incidents/
├─ hooks/
├─ lib/
├─ pages/
│  ├─ ConsoleOverviewPage.tsx
│  ├─ ProjectIntakePage.tsx
│  ├─ ParsePipelinePage.tsx
│  ├─ AssetPoolPage.tsx
│  ├─ ProjectDetailPage.tsx
│  ├─ EditingWorkbenchPage.tsx
│  ├─ ReviewWorkbenchPage.tsx
│  ├─ DeliveryPage.tsx
│  ├─ OpsConfigsPage.tsx
│  └─ IncidentCenterPage.tsx
└─ types/
   ├─ production.ts
   ├─ project.ts
   ├─ incident.ts
   ├─ review.ts
   └─ config.ts
```

## 6.2 为什么这样拆

- `pages/` 负责路由页面容器
- `features/` 负责页面领域逻辑
- `components/production` 放工业后台专用组件
- `types/` 放统一领域类型

这样后续真开发时不会把每页写成 500 到 800 行大文件。

但从 V5 开始建议统一 query key 规范。

建议：

```text
['console', 'overview']
['intake', 'summary']
['intake', 'projects', filters]
['parse', 'tasks', filters]
['assets', 'projects', filters]
['project', projectId, 'summary']
['project', projectId, 'versions']
['review', 'queue']
['delivery', 'packages']
['delivery', 'feedback']
['incidents', 'list', filters]
['incidents', 'summary']
['configs', 'providers']
```

这样后面做批量刷新、局部失效、轮询就很清楚。

## 13. 开发顺序建议

为了尽快让页面真的跑起来，建议按照下面顺序做。

## 阶段 1：先搭壳层

- 升级 `Layout.tsx`
- 抽出 `AppShell`
- 做 V5 左侧导航
- 做 `GlobalStatusStrip`

## 阶段 2：先搭 4 个最关键页面

优先做：

1. `生产总控台`
2. `AI配置与运维中心`
3. `异常中心`
4. `人工快审工位`

原因：

- 最能体现 V5 风格
- 现有接口复用最多
- 最快产出“像真系统”的感觉

## 阶段 3：再补项目闭环

- 项目入库检验台
- 解析作业调度台
- 短剧生产资产池
- 项目调度详情台

## 阶段 4：最后补生产终端

- 批量版本编排工位

## 14. 建议新增的后端资源语义

这一节只定义资源语义，不锁死字段细节。

建议新增这些资源组：

- `projects`
- `assets`
- `editing`
- `delivery`
- `incidents/root-cause`

建议原因：

- 当前后端已经有账号、队列、剧源、配置、异常、复核等资源
- 但还缺少真正面向 `短剧项目` 和 `版本生产` 的资源层

这也是为什么 V5 页面如果要真开发，必须新增一部分面向“项目资产”的接口。

## 15. 开发完成标准

V5 前端开发准备完成的标准，不是“文档写完”，而是下面这些条件满足：

1. 新路由已经能跑通
2. V5 布局壳层已经替换旧壳层
3. 公共工业组件已经抽出来
4. 至少 4 个页面已接真实接口
5. 页面不再依赖大块硬编码 demo 数据
6. `项目`、`版本`、`异常`、`配置` 这四类核心类型已统一

## 16. 一句话结论

V5 现在已经不适合继续只看效果图了。  
它应该进入下面这个阶段：

`保留现有 React + FastAPI 基座，新增 V5 页面层、工业组件层、项目资源层，然后分阶段替换旧控制台页面。`

如果继续往下推进，下一步最值钱的动作不是再出图，而是：

`先把 V5 的前端目录骨架和路由骨架搭出来。`
