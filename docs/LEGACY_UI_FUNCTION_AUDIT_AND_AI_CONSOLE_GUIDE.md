# 旧版人工界面功能审计与 AI 控制台继承建议

## 1. 文档定位

本文档基于你提供的旧版“火视界短剧”截图，对原软件做能力审计。

重点不是照搬旧界面，而是回答 3 个问题：

- 旧软件已经沉淀了哪些真实业务能力
- 哪些能力值得继承，哪些只适合降级为低频配置
- 我们的 AI 自动控制系统还需要补哪些“界面端配置能力”

结论先说：

- 旧软件适合人工批量操作，不适合直接作为新系统的信息架构
- 新系统必须从“人工操作台”升级成“AI 控制台 + 策略中台 + 观测中台 + 人工接管台”
- 旧软件里的很多功能要保留，但要从“大表单”改成“模板、策略、开关、护栏、例外处理”

## 3. 我对旧界面的核心判断

旧软件的问题不在于“没功能”，而在于“功能有了，但组织方式很不适合生产级 AI 系统”。

### 3.1 可以直接继承的，是能力，不是布局

值得继承的不是它的双栏大表单，而是这些能力：

- 剧源提取与分配
- 采集条件与去重条件
- 发布通道切换
- 视频处理策略
- 账号登录与 Cookie 运维
- 收益与机构结算视图

### 3.2 旧界面最大的问题是“把所有低层参数暴露给人工”

例如这些内容，在旧软件里全部堆在主流程页面：

- MD5 修改
- 3:4 前置处理
- 去重算法模式
- 字体与水印位置
- CPU/GPU 模式
- 浏览器上传 / API 上传
- 权限、互动、作者声明
- 文件路径

这些并不是“日常运营动作”，而是：

- 模板参数
- 高级参数
- 引擎参数
- 调优参数

它们不应该占据主操作区。

### 3.3 它是人工执行逻辑，不是 AI 调度逻辑

旧软件的核心结构是：

- 人工选配置
- 人工点执行
- 人工看状态

而我们的新系统应该是：

- AI 决策生成计划
- 系统自动派发任务
- 人工只看摘要、看异常、做少量干预

所以新系统界面必须围绕：

- 目标
- 策略
- 模板
- 护栏
- 执行状态
- 结果归因
- 人工接管

来设计，而不是围绕“按钮和参数”。

## 4. 旧软件里不合理的地方

### 4.1 首页式大杂烩

“账号与短剧模式”一页里混了：

- 剧源提取
- 账号选择
- 分配逻辑
- 平台模式
- 并发设置
- 视频处理
- 发布参数
- 执行按钮

这是典型的“能做，但不可管”。

### 4.2 配置、执行、监控没有分层

旧软件把三种事情混在一起：

- 一次性配置
- 日常运营
- 实时执行

结果就是：

- 页面非常长
- 学习成本极高
- 误操作风险大
- 很难做 AI 接管

### 4.3 设备视角和账号视角混在一起

截图里同时出现：

- 设备列表
- 浏览器账号
- `no_device`
- `web_xxx`

这说明旧系统没有把“资源层”梳理干净。

新系统必须明确区分：

- 设备资源
- 浏览器账号
- 平台账号
- MCN 绑定成员

### 4.4 任务队列和任务记录割裂且重复

旧软件里：

- 任务队列看实时
- 任务记录看历史

但结构相似、字段相近、心智重复。

新系统应统一为：

- 批次中心
- 任务中心
- 任务详情抽屉
- 历史筛选只是同一页面的一个视图

### 4.5 收益查询与账号运营链路脱节

旧软件收益页能看收益，但没有打通：

- 账号
- 作品
- MCN 绑定
- 邀约状态
- 结算凭证
- 放大决策

这对 AI 系统来说不够，因为 AI 要能把收益数据回流到策略。

### 4.6 顶部快捷工具太多，且没有主次

例如：

- 扫描设备
- 抖音 Cookie
- 导入配置
- 导出配置
- 导入收藏
- 导出收藏

这些适合放到：

- 系统工具箱
- 数据导入导出中心
- 资源中心

不适合常驻主导航上方。

## 5. 对我们现有规划的补充建议

你的判断是对的。

我们虽然是 AI 自动控制系统，但界面端必须保留“相应配置”，否则 AI 没法被约束，也没法被运营人员调整。

只是这些配置不能继续做成旧软件那种大表单，而要做成分层控制台。

我建议把界面配置拆成 5 层。

## 5.1 L1 一次性初始化配置

低频、管理员配置：

- 设备接入
- 账号接入
- Cookie / 授权初始化
- MCN 机构接入
- `owner_code`
- WebSocket / heartbeat 参数
- 存储路径
- GPU / FFmpeg / 浏览器环境
- 导入导出配置

特点：

- 不属于日常运营
- 应收进“系统配置中心 / 资源中心”

## 5.2 L2 业务模板配置

中频配置，主要给运营使用：

- 采集模板
- 剧源提取模板
- 视频处理模板
- 水印模板
- 发布模板
- 互动权限模板
- 标签模板
- 账号分组模板

特点：

- 不直接改底层参数
- 以模板复用替代表单重复填写

## 5.3 L3 AI 策略配置

这是旧软件没有、但我们必须有的：

- AI 目标权重
- 收益优先 / 起号优先 / 稳定优先
- 探索比例
- 放大倍率上限
- 样本量门槛
- 实验成功阈值
- 自动扩量条件
- 自动停损条件
- 热剧跟进策略
- 账号阶段迁移规则

特点：

- 这是 AI 自动系统的核心配置层
- 应该独立做“AI 控制中心 / 策略中心”

## 5.4 L4 运行护栏配置

这是 AI 可执行但必须被约束的边界：

- 每日发布上限
- 单账号间隔
- 禁发时间段
- 风险账号禁止自动发
- MCN 未绑定禁止收益发布
- 邀约待确认禁止放量
- 连续失败熔断
- 连续低播放降量
- 人工审核触发阈值

特点：

- 本质上是“系统安全阀”
- 应放在“开关与护栏中心”

## 5.5 L5 高级引擎配置

这是技术人员低频配置：

- 去重算法
- 3:4 前处理
- MD5 处理
- CPU / GPU 模式
- 上传通道优先级
- Worker 并发
- 失败重试策略
- Provider 切换
- Prompt 版本
- 学习周期

特点：

- 绝不能默认暴露给普通运营
- 默认折叠或放到“高级设置”

## 7. 新控制台建议新增的页面或模块

如果按 AI 自动控制系统来做，我认为我们现有规划还应更明确下面几个页面。

## 7.1 AI 总控中心

不是看日志，而是看：

- 今日目标
- AI 计划摘要
- 阻断原因
- 当前实验
- 当前放量
- 当前止损
- 计划执行进度

## 7.2 模板中心

专门管理：

- 采集模板
- 处理模板
- 发布模板
- 账号模板
- 水印模板

这能把旧软件那种超长配置表单拆掉。

## 7.3 护栏中心

专门管理：

- 开关
- 风险阈值
- 发布边界
- MCN 门禁
- 自动停损
- 人工审核触发器

## 7.4 系统工具箱

收纳低频操作：

- 扫描设备
- 导入导出配置
- 导入导出收藏
- 文件锁清理
- 浏览器环境修复
- 重建连接

## 7.5 收益与结算中心

不只是查收益，还要打通：

- 账号
- 成员
- 机构
- 邀约
- 绑定
- 作品
- 收益
- 分成
- 结算凭证

---

## 8. 对当前规划的最终补充判断

当前我们的总体架构方向是对的，但界面规划还可以再补强 4 个点：

1. 现在“开关与配置中心”还偏轻，需要升级成“系统配置 + 护栏 + 高级设置”三层结构。
2. 现在“AI 决策中心”更像看结果，还需要补“AI 目标配置 / 策略权重 / 放量规则 / 止损规则”。
3. 现在“账号中心”需要更明确区分“账号运营”和“MCN / 结算 / 绑定”。
4. 现在还缺“模板中心”，否则旧软件里那些大量参数最后还是会重新堆回主页面。

---

## 9. 最终建议

旧软件给我们的价值，不是 UI 参考，而是业务能力盘点。

新系统界面应该遵循这个原则：

- 高频动作前置
- 低频配置收纳
- AI 决策可解释
- 人工接管有入口
- 模板替代表单
- 护栏约束自动化

如果按这个方向走，我们新系统会和旧软件拉开本质差距：

- 旧软件是“人工批处理工具”
- 新系统是“AI 自动运营控制台”
