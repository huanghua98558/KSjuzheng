# 快手短剧AI全自动运营系统 — 完整架构设计 V2.0

## 一、系统定位

**不是脚本集合，是完整的AI运营平台。**

核心逻辑闭环：
```
获取短剧 → 智能选剧 → 下载视频 → 自动剪辑去重 → 智能决策发布 →
数据回收分析 → 反馈优化（选剧+去重+发布策略） → 放大/淘汰 → 循环
```

## 二、系统分层架构（8层）

```
┌──────────────────────────────────────────────────────┐
│  L8  监控仪表盘 + 配置中心                              │
│      Web UI / 自动化配置面板 / 微信推送 / Excel报表       │
│      初期: 调试参数面板  中期: 一键稳定运行               │
├──────────────────────────────────────────────────────┤
│  L7  数据分析层                                        │
│      收益/账号/短剧/视频/去重效果 多维分析                │
├──────────────────────────────────────────────────────┤
│  L6  智能决策层                                        │
│      AI选剧 / 去重策略优化 / 发布策略 / 起号方案          │
├──────────────────────────────────────────────────────┤
│  L5  执行引擎层                                        │
│      任务队列 / 重试 / 降级 / 熔断 / 并发控制            │
├──────────────────────────────────────────────────────┤
│  L4  业务逻辑层                                        │
│      短剧获取 / 下载 / 剪辑去重 / 发布 / 测试(A/B)      │
├──────────────────────────────────────────────────────┤

├──────────────────────────────────────────────────────┤

## 三、L4 核心业务模块详解

### 4.1 短剧获取与分析子系统

┌─ 短剧源管理器 ──────────────────────────────────────┐
│                                                      │

│  数据源2: 快手推荐feed (feed/selection)                 │
│  数据源3: 达人主页采集 (profile/feed)                   │
│  数据源4: 手动导入 (URL/Excel批量)                      │
│  数据源5: 分享链接解析 (share/w/any)                    │
│  数据源6: 竞品账号监控 (定期抓取头部账号在推什么剧)       │
│                                                      │
├─ 短剧分析器 ────────────────────────────────────────┤
│  - 短剧热度评分（播放量/点赞/评论/分享）                 │
│  - 同行发布数据（有多少人在推这个剧）                    │
│  - 转化率预估（历史数据回归）                            │
│  - 竞争饱和度分析                                      │
│  - 剧目生命周期判断（上升期/稳定期/衰退期）              │
│                                                      │
├─ 剧库管理 ──────────────────────────────────────────┤
│  - drama_links表: 待发布剧目池                          │
│  - download_cache表: 已缓存剧目                         │
│  - 自动去重: 同一剧不重复入库                            │
│  - 智能分配: 按账号赛道/等级匹配合适的剧                 │
│  - 淘汰机制: 低效剧自动下架                             │

└──────────────────────────────────────────────────────┘
```

#### 配置项（从原软件GUI捕获）：
- `video_processing_method`: mode3/mode6/interleave/overlay/maker/zhizun/rongyu
- `video_processing_random`: 是否随机选择模式
- `overlay_opacity`: 叠加透明度 (默认0.3)
- `overlay_animations`: 叠加动画效果列表
- `drama_hide_template_tag`: 隐藏模板标记
- `drama_hide_kuaiying_tag`: 隐藏快影标记
- `drama_replace_mode`: 视频替换模式 (rename_to_shortest / copy_to_all)

#### 去重架构设计：

```
┌─ 视频处理引擎 ──────────────────────────────────────┐
│                                                      │
│  ┌─ 策略注册表 ────────────────────────────────┐     │
│  │  每种去重模式注册为独立Strategy类              │     │
│  │  统一接口: process(input, output, config)     │     │
│  │  每个Strategy报告自己的资源需求和耗时预估      │     │
│  └──────────────────────────────────────────────┘     │
│                                                      │
│  已实现策略（复用原软件）:                              │
│  ├─ Mode3Strategy         基础去重                    │
│  ├─ Mode3EnhancedStrategy 增强去重                    │
│  ├─ MakerStrategy         创作者模式                  │
│  ├─ OverlayStrategy       画面叠加                    │
│  ├─ InterleaveStrategy    帧交错                      │
│  ├─ Mode6Strategy         九宫格混合 ✅               │
│  ├─ InterleaveEnhancedStrategy 帧交错增强             │
│  ├─ Scale34Strategy       3:4比例变换                 │
│  ├─ ZhizunStrategy        至尊版(GPU加速)             │
│  └─ RongyuStrategy        荣誉版(OpenGL)              │
│                                                      │
│  我们新增策略:                                        │
│  ├─ ColorShiftStrategy    色彩微调(HSV偏移)           │
│  ├─ SpeedJitterStrategy   微变速(0.98-1.02x)          │
│  ├─ CropPadStrategy       裁切+边距                   │
│  ├─ MetadataStrategy      元数据重写                  │
│  └─ CompositeStrategy     多策略随机组合               │
│                                                      │
│  ┌─ 去重效果追踪器 ──────────────────────────────┐   │
│  │  每次处理记录: 视频ID + 策略名 + 参数哈希        │   │
│  │  发布后追踪: 该视频是否通过审核、播放量如何       │   │
│  │  反馈到L6决策层: 哪种策略通过率最高              │   │
│  │  自动淘汰: 通过率低于阈值的策略自动降权          │   │
│  └──────────────────────────────────────────────┘   │
│                                                      │
│  GPU管理:                                            │
│  ├─ h264_nvenc (CUDA) → 首选                         │
│  ├─ libx264 (CPU) → 降级方案                         │
│  └─ OpenGL (bin3/) → 荣誉版专用                      │
│                                                      │
│  缓存: video_processing_cache表                       │
│  ├─ 同一视频+同一策略 = 不重复处理                    │
│  └─ 不同账号发同一剧 = 必须用不同策略/参数            │
└──────────────────────────────────────────────────────┘
```

### 4.4 发布子系统（双通道 + 自动降级）

```
┌─ 发布引擎 ──────────────────────────────────────────┐
│                                                      │
│  通道A（HTTP API）—— 快速高效                          │
│    复用: 逆向捕获的完整API流程                          │
│    upload/pre → fragment → complete →                  │
│    finish → banner → submit                            │
│    优点: 快(30s完成)、可并发、无需浏览器                 │
│    缺点: 需要有效Cookie+签名，可能被风控                │
│                                                      │
│  通道B（Selenium浏览器）—— 稳定保底                     │
│    复用: 原软件KuaishouWebPublisher（成熟代码）          │
│    通过内嵌Chrome操作cp.kuaishou.com                    │
│    优点: 模拟真人操作，不易被风控                        │
│    缺点: 慢(3-5分钟)、不能并发                          │
│                                                      │
│  通道切换策略:                                         │
│  ├─ 默认: A优先                                       │
│  ├─ A失败 → 自动降级到B                                │
│  ├─ 连续A失败3次 → 该账号临时切到B模式(30分钟)          │
│  ├─ 可按账号/时段/风控等级手动配置                      │
│  └─ 仪表盘一键切换全局默认通道                          │
│                                                      │
│  发布后处理:                                           │
│  ├─ 验证发布成功(检查作品列表)                          │
│  ├─ 获取share_link写入数据库                            │
│  ├─ 挂载变现任务(spark/firefly)                         │
│  └─ 记录execution_log                                  │
└──────────────────────────────────────────────────────┘
```

### 4.5 测试板块（A/B Testing + 放大机制）

```
┌─ 测试管理器 ────────────────────────────────────────┐
│                                                      │
│  Step 1: 创建测试批次                                  │
│    选N个视频/剧目 → 分配到M个测试账号                    │
│    每个剧用2-3种不同去重策略各发一版                     │
│    → 发布 → 标记为"测试批次#xxx"                        │
│                                                      │
│  Step 2: 数据回收 (24h/48h/72h三个时间点)               │
│    自动获取每个作品的:                                   │
│    - 播放量/完播率/点赞率/评论率/分享率                  │
│    - 收益(每万播放收益 = CPM)                           │
│    - 粉丝增长                                          │
│    - 审核通过率（不同去重策略的通过率对比）               │
│                                                      │
│  Step 3: 分析决策                                      │
│    效果好(CPM>阈值) → 标记为"高转化"                    │
│      → 放大: 分配到全部账号 + 使用通过率最高的去重策略   │
│    效果差(CPM<阈值) → 标记为"低效"                      │
│      → 淘汰: 停止投放该剧                               │
│    效果中 → 换一种去重策略重试                           │
│                                                      │
│  Step 4: 反馈闭环                                      │
│    测试结果 → 更新剧目评分 → 影响智能选剧               │
│    去重通过率 → 更新策略权重 → 影响去重策略选择          │
└──────────────────────────────────────────────────────┘
```

## 四、L5 执行引擎层

```
┌─ 任务调度引擎 ──────────────────────────────────────┐
│                                                      │
│  任务类型 + 资源需求:                                  │
│  ┌────────────┬──────────┬──────────┬────────────┐  │
│  │ 类型        │ 耗时      │ 并发上限  │ 资源       │  │
│  ├────────────┼──────────┼──────────┼────────────┤  │
│  │ DOWNLOAD   │ 1-5 min  │ 3       │ 网络带宽    │  │
│  │ PROCESS    │ 1-10 min │ 2(GPU)  │ GPU/CPU    │  │
│  │ PUBLISH_A  │ 30s      │ 5       │ 网络       │  │
│  │ PUBLISH_B  │ 3-5 min  │ 1       │ Chrome     │  │
│  │ COLLECT    │ 10-30s   │ 5       │ 网络       │  │
│  │ ANALYZE    │ 5-30s    │ 10      │ CPU        │  │
│  └────────────┴──────────┴──────────┴────────────┘  │
│                                                      │
│  任务依赖链:                                          │
│  COLLECT → DOWNLOAD → PROCESS → PUBLISH → ANALYZE    │
│  (每个环节可独立失败重试，不影响其他环节)               │
│                                                      │
│  错误处理层次:                                         │
│  ├─ L1 重试: 同一方式重试 (3次, 指数退避)              │
│  ├─ L2 降级: 换备选方案 (GPU→CPU, API→Selenium)        │
│  ├─ L3 熔断: 连续5次失败 → 暂停账号30分钟              │
│  ├─ L4 告警: 通知人工干预                              │
│  └─ L5 跳过: 标记失败,继续下一个任务                    │
│                                                      │
│  调度策略:                                             │
│  ├─ 时间窗口: 发布集中在6:00-23:00                     │
│  ├─ 发布间隔: 同账号间隔5-15分钟(可配置)               │
│  ├─ 批量预处理: 夜间批量下载+剪辑，白天定时发布         │
│  └─ 优先级: 紧急>高转化测试>常规发布>数据采集           │
└──────────────────────────────────────────────────────┘
```

## 五、L6 智能决策层

```
┌─ AI决策引擎 ────────────────────────────────────────┐
│                                                      │
│  ┌─ 选剧决策 ────────────────────────────────────┐  │
│  │  输入: 剧库列表 + 历史数据 + 竞争分析            │  │
│  │  输出: 每个账号今天推什么剧                      │  │
│  │  规则:                                          │  │
│  │    新号 → 选热门剧快速起量                       │  │
│  │    老号 → 选高转化剧优化收益                     │  │
│  │    避免同账号重复推同一剧                        │  │
│  │    避免所有账号推同一剧(分散风险)                 │  │
│  │    考虑剧的生命周期(优先上升期的剧)               │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
│  ┌─ 去重策略决策 ────────────────────────────────┐  │
│  │  输入: 各策略历史通过率 + 各策略收益表现          │  │
│  │  输出: 每个视频用哪种去重策略                    │  │
│  │  规则:                                          │  │
│  │    同一剧不同账号 → 必须用不同策略               │  │
│  │    通过率高的策略 → 权重提高                     │  │
│  │    新策略 → 小比例探索性使用                     │  │
│  │    连续被拒 → 自动切换策略                       │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
│  ┌─ 发布决策 ────────────────────────────────────┐  │
│  │  输入: 账号等级 + 当日已发 + 时间段             │  │
│  │  输出: 发几条、什么时候发、用哪个通道            │  │
│  │  规则:                                          │  │
│  │    V1(1-3天): 7条/天 (冷启动期)                 │  │
│  │    V1(4天+): 3条/天 (稳定运营)                  │  │
│  │    V3: 5条/天                                   │  │
│  │    V4+: 20条/天                                 │  │
│  │    风控感知: 失败率上升 → 自动降速               │  │
│  └──────────────────────────────────────────────┘  │
│                                                      │
│  ┌─ 起号策略 ────────────────────────────────────┐  │
│  │  新号冷启动方案:                                │  │
│  │    Day 1-3: 高频发布(7条) + 热门剧              │  │
│  │    Day 4-7: 数据分析 → 调整方向                  │  │
│  │    Day 8+:  稳定运营 / 放弃换号                  │  │
│  │  测试批次 → 数据分析 → 效果好放大 / 效果差换策略  │  │
│  └──────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```
