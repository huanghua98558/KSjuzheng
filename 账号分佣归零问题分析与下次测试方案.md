## 3. 我之前的判断是错的

### 3.1 promotion_type 分类真相 (180° 相反)

| promotion_type | 我之前判断 | 实际情况 |
|---|---|---|
| `None` / 空 | "废弃任务, 不分佣" | **老萤光任务, 当前真正在产生收益的主力** |
| `7` (效果计费) | "最新活跃任务" | **新任务, 大部分还 0 元收益** |

### 3.2 "财源滚滚小厨神" 同名 7 个 biz_id 收益对照

| biz_id | promotion_type | 收益条数 | 累计 |
|---|---|---|---|
| 170767 | None | 79 | **¥334.10** |
| **169570** | **None** | **33** | **¥19.33** |
| 169598 | None | 0 | 0 |
| 164594 | None | 0 | 0 |
| 87708 | None | 0 | 0 |
| 3845204 | 7 (效果计费) | **0** | **0** ← 我让用户改挂这个 |
| 3912934 | 7 (效果计费) | **0** | **0** |

**教训**: "新上线 + 效果计费 + view=1 + end_time 未来" 不等于"会产生收益". 老任务虽然看起来字段空, 但**运行几个月了, 有真实播放数据, 才是真实的分佣池**.

### 3.3 `sync_mcn_drama_library.py` 过滤策略必须改

原过滤 (错的):
```sql
WHERE promotion_type = 7 AND view_status = 1
  AND (end_time IS NULL OR end_time = 0 OR end_time > UNIX_TIMESTAMP())
```

上面这条把所有实际产生收益的老任务**全过滤掉了**. 目前本地 `drama_banner_tasks` 表里 4036 个任务几乎都是"还没开始赚钱的新任务".

**新策略** (下一步要改):
```sql
-- 保留 "过去 30 天在 fluorescent_income 有分佣记录" 的任务
-- 同时保留 "新任务, 即使 promotion_type=7 也允许"
FROM spark_drama_info d
LEFT JOIN (
  SELECT task_id, COUNT(*) n, SUM(income) s, MAX(created_at) last_income
  FROM fluorescent_income
  GROUP BY task_id
) i ON CAST(d.biz_id AS CHAR) = i.task_id
WHERE i.last_income >= DATE_SUB(NOW(), INTERVAL 30 DAY)
   OR (d.promotion_type = 7 AND d.view_status = 1
       AND (d.end_time IS NULL OR d.end_time = 0 OR d.end_time > UNIX_TIMESTAMP()))
```

把"真实分佣历史"放第一位, 新任务只作补充.

## 4. 修正: "加入任务"不是缺失的环节

### 4.1 MCN MySQL 50 张表审查结论

- `fluorescent_income` (29,472 行, 按 member+task+创建时间) — 事件流, 有延迟
- `fluorescent_income_archive` (12,489 行, 按月存档) — 不含 task_id
- **`fluorescent_members` (16,772 行) — 实时成员状态, 含 `total_amount` 和 `org_task_num`** ← 最权威
- `spark_members` / `spark_org_members` — 星火计划专用, 荧光不走这里
- `firefly_members` (218 行) — 萤光计划, 很少有账号签

### 4.2 账号 3 其实已"加入" 1 个任务

`fluorescent_members` 里:
- `org_task_num=1` 就是"当前加入的任务数"
- `total_amount=¥0.07` 就是"今日累计收益"
- **不需要我们主动调"加入任务"API** — 系统已经识别我们发的视频挂了某个 task_id

### 4.3 MCN 前端爬虫确认

扫描 MCN 后台 SPA (`index-BXe01rQ4.js` + 25 个 lazy-chunks) 发现:

- 唯一"加入"按钮是 `FluorescentIncome.js` + `SparkIncome.js` 里的 **「加入高转化」**
- 这是"把某个 task_name 加到 `spark_highincome_dramas` 高收益清单" 的机构管理动作
- **不是**"让某账号加入某任务"的动作
- MCN 后台没有"给账号加入任务"的主动操作入口

### 4.4 真相推测

```
视频发 publisher → bannerTaskId=XXX in payload → 快手后端收录
    ↓
  快手内部: "这个视频有 task_id=XXX, 属于 member_id=YYY"
    ↓
  每日批处理: 按播放量产生 income, 写进快手收益表
    ↓
  MCN 同步快手数据: fluorescent_members 表 refresh
    ↓
  MCN 前端看板展示 (我们今天看到的 ¥0.07)
```

关键: **只要视频 payload 带对的 bannerTaskId + 视频真的发出去, 快手自己会把视频归到这个 task_id 下**. 我们不需要"先加入任务再发视频".

**真正缺的是**:
1. publisher.py 假-success bug (已修) 导致我们以为发了其实没发
2. bannerTaskId 选错 (挂 167452 / 169570 这种低产出任务) 导致即使发出去收益也少
3. 对视频投放的"质量控制"不够 (caption / 封面 / 发布频率) 影响播放量

## 9. 关键技术事实沉淀

### 9.1 MCN MySQL 4 张收益表的角色

| 表 | 粒度 | 何时产生记录 |
|---|---|---|
| `firefly_income` (萤光, 3448 行) | 每个视频 | 视频有播放 + 属于萤光计划任务 |
| `fluorescent_income` (荧光, 29472 行) | 每个 (member, task) 对 | 账号加入任务 + 有播放贡献 |
| `spark_income` (星火, 80 行) | 星火计划专用 | 星火计划成员 + 视频有收益 |
| `spark_income_archive` (3307 行) | 星火月度存档 | 每月 1 号结算后写入 |

**一张票出错 → 收益归零**.

### 9.2 账号 ID 对照

- MCN 的 `member_id` = **数字 UID** (`887329560`)
- 数字 UID 从 cookie `userId=` 字段提取

### 9.4 关键 SQL 片段

**查某账号实际分佣 (全部 4 张表)**:
```sql
-- 如果账号 MCN member_id = $MID
SELECT 'firefly' AS src, SUM(commission_amount) AS total
FROM firefly_income WHERE author_id = '$MID'
UNION ALL
SELECT 'fluorescent', SUM(income) FROM fluorescent_income WHERE member_id = $MID
UNION ALL
SELECT 'spark', SUM(income) FROM spark_income WHERE member_id = $MID
UNION ALL
SELECT 'spark_archive', SUM(commission_amount)
FROM spark_income_archive WHERE member_id = $MID;
```

**查某任务谁在赚**:
```sql
SELECT member_name, COUNT(*) n, SUM(income) s
FROM fluorescent_income
WHERE task_id = '$TASK_ID'
GROUP BY member_name ORDER BY s DESC;
```
